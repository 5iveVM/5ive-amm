# Five CLI / MitoVM Runtime Failure — Investigation Plan (read-only)

Path: five-org/five-cli/INVESTIGATION_PLAN.md
Purpose: A stand-alone, investigation-only plan other agents can follow to reproduce, triage, and gather artifacts for the InvalidVariableIndex runtime failures seen in the 01-language-basics tests. This document intentionally avoids making or applying fixes — it's designed so multiple investigators can run different probes in parallel and report findings.

---
Status summary (short)
- Symptom: Several 01-language-basics scripts fail at runtime with MitoVM errors such as `InvalidVariableIndex(Some(N))` even though compilation succeeds.
- Evidence: Compiled `.five` outputs show `ALLOC_LOCALS`, `LOAD_PARAM`, `SET_LOCAL`, `GET_LOCAL` opcodes. In many cases ALLOC counts appear to cover the highest GET/SET indices, yet runtime errors persist during nested-call execution paths.
- Likely class of problem: A runtime call-frame / parameter-window / local-window mismatch between what the compiler emits and what the VM expects at the moment of a GET_LOCAL or SET_LOCAL (ordering/timing or off-by-one in frame math).

Goal for investigators
- Reproduce the failure locally.
- Collect authoritative artifacts (compiled `.five`, raw bytecode, opcode maps, VM traces).
- Produce evidence to precisely identify whether the mismatch is:
  - Compiler emission ordering (GET_LOCAL emitted prior to ALLOC/param-preload),
  - VM call-frame setup bug (CALL handler sets local_base/local_count wrong),
  - ABI / parameter transfer mismatch (CALL parameter window vs LOAD_PARAM semantics),
  - Or a combination.
- Do not make code changes yet — collect data so multiple opinions can be formed. If you find a clear, minimal fix, stop and propose the patch rather than applying it.

  - node -e "const fs=require('fs');const j=JSON.parse(fs.readFileSync('/tmp/integration-complex.five','utf8'));const b=Buffer.from(j.bytecode,'base64');for(let i=0;i<b.length;i++){ if([0xA0,0xA2,0xA3,0xA5].includes(b[i])) console.log(i,b[i].toString(16),b[i+1]); }"
- Run a single function with the local wasm VM:
  - node dist/index.js --verbose local execute test-scripts/01-language-basics/integration-complex.v 0

Investigation tasks (parallel roles)
- Role 1 — Compiler Investigator
  - Task C1: Confirm exact compiler emission order for each function:
    - Verify: `ALLOC_LOCALS` appears before any `SET_LOCAL` or `GET_LOCAL` inside that function body.
    - Verify: parameters are loaded (LOAD_PARAM X) and then stored into locals (SET_LOCAL Y) before other GET_LOCAL ops reference them.
    - Where to look: `function_dispatch.rs` → `generate_single_function_body`, `ast_generator/*` for identifier emission.
  - Task C2: Confirm AST symbol table offsets:
    - After the compiler runs (you can instrument or print), list `ast_generator.local_symbol_table` entries for a function: name → offset, is_parameter.
    - Ensure offsets assigned for locals do not change after preloading params and before GET_LOCAL emissions.
  - Task C3: Produce per-function opcode map from `.five` (ALLOC/SET/GET/CALL offsets & args) and attach to artifacts.

- Role 2 — VM Investigator
  - Task V1: Reproduce failure and capture VM frame state on CALL + on failing GET/SET:
    - Add conservative TRACE prints (if allowed for investigation) in `handlers/functions.rs` near CALL and in `handlers/locals.rs` on GET/SET: include IP, param_start, param_len, local_base, local_count, SP, call_depth.
    - Re-run failing test to capture the trace. Do NOT alter VM semantics; only log.
    - Goal: see whether the CALL set the callee local_count/local_base as expected, and whether GET_LOCAL occurs before locals are ready.
  - Task V2: Validate CALL parameter transfer:
    - Confirm CALL pops arguments from caller stack and writes them to `parameters[]` with the expected index mapping (1-based vs 0-based).
    - Confirm callee’s GET_PARAM / LOAD_PARAM semantics match the compiler's emitted `LOAD_PARAM` numbers.

- Role 3 — Test/Artifact Collector
  - Task T1: Run the test-runner and collect full logs for all failing scripts (use `--verbose` and `--errors-only` flags).
  - Task T2: For each failing script, run the compile-and-extract pipeline (create `.five`, binary, opcode map) and attach results.
  - Task T3: Reduce a failing script to the minimal reproducer and include the reduced `.five` + bytecode map.

Prioritized hypothesis checks (fast order)
1. Emission ordering mismatch (compiler): check ALLOC_LOCALS location vs earliest GET_LOCAL in same function body. If GET_LOCAL comes earlier in the bytecode than ALLOC_LOCALS, compiler ordering is wrong.
2. AST offsets not shifted: check that parameters were registered as locals or that field_counter was advanced so subsequent locals don't overlap parameter slots.
3. CALL frame mismatch (VM): if ALLOC_LOCALS exists in bytecode before GET_LOCAL but at runtime the VM's local_count is zero at callee entry, then CALL setup or parameter transfer is the likely problem.
4. Immediate/metadata interference: check if compiler is emitting metadata between CALL and the function prologue (e.g., name bytes) that confuse VM when it expects ALLOC immediately. The VM should skip metadata but verify skip logic.
5. Edge-cases: nibble immediates (0xD0..0xDF) vs full opcodes — ensure both compiler and VM use same ranges and naming.
