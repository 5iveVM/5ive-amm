4. Result: Parameters allocated to registers based on optimal live range analysis

## Architecture & Design

### Data Flow

```
Five DSL Source
    ↓
Parser & Type Checker
    ↓
Scope Analyzer (computes first_use, last_use, usage_count)
    ↓
Function Dispatch (if --enable-registers --use-linear-scan)
    ↓
LiveIntervalBuilder (converts scope → live intervals)
    ↓
LinearScanAllocator (allocates registers)
    ↓
ASTGenerator (uses register allocations)
    ↓
Optimized Bytecode
```

### Register Allocation Algorithm

**Time Complexity:** O(n log n) - dominated by sorting intervals
**Space Complexity:** O(n) - for interval storage

**Steps:**
1. Sort intervals by start position (O(n log n))
2. For each interval in order:
   - Remove expired intervals from active set
   - Find free register from available registers
   - Assign or mark for spilling
3. Return register → variable mappings

**Features:**
- ✅ Register reuse when variables don't overlap
- ✅ Spilling support for >16 variables
- ✅ Parameter priority (always allocated first)
- ✅ Usage-aware allocation (high-use vars get better registers)

## Known Limitations & Future Work

### Current Limitations
1. **Spilling** - Variables exceeding 16 registers are marked but not spilled to stack
2. **Furthest-Use Heuristic** - Not yet implemented for spill selection
3. **Loop Detection** - No special handling for loop-bound variables
4. **Priority Hints** - No account-aware or Five-specific heuristics
